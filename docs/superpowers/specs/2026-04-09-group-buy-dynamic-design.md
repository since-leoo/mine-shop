# 拼团专题页动态数据设计

## 背景

当前拼团专题页位于 `miniprogram/src/pages/promotion/group-buy/index.tsx`。页面视觉结构已经接近设计稿，但核心运营区块仍包含较多静态文案和静态指标，导致页面“像专题页，但数据不可信”。

本次目标是基于现有接口，把用户指出的 3 个重点区域改为真实动态数据驱动，同时补齐必要的交互和空态兜底，避免继续依赖写死内容。

## 目标

- 将“拼团氛围卡”改为真实数据展示
- 将“还有 N 个团正在冲刺成团”改为真实进行中团数据驱动
- 将“主推团长款”中的价格、已成团、缺口、成功率、按钮行为改为动态
- 保持当前页面视觉方向，不做无关重构
- 在接口字段不足时进行前端安全降级，确保页面稳定

## 非目标

- 不新增后端接口
- 不重做页面整体视觉结构
- 不引入新的全局状态管理
- 不实现“直接指定某个 groupNo 参团”的新下单链路，除非现有详情页已天然支持

## 现有数据源

### 1. 拼团商品列表

来源：`fetchGroupBuyList()`

用途：

- 拼团专题页商品列表
- 主推商品（默认取当前筛选结果第一条）
- 导航分组回退逻辑
- 汇总型指标推导

当前已能读取的字段包括但不限于：

- `spuId`
- `activityId`
- `thumb`
- `title`
- `price`
- `originPrice`
- `minPeople`
- `groupCount`
- `successGroupCount`
- `groupTimeLimit`
- `tags`
- `soldQuantity`

### 2. 进行中团列表

来源：`fetchOngoingGroups(activityId, limit)`

用途：

- “还有 N 个团正在冲刺成团”
- 主推团长信息
- 还差几人成团
- 剩余时间提示

当前已映射字段包括：

- `groupNo`
- `leaderNickname`
- `leaderAvatar`
- `joinedCount`
- `needCount`
- `expireTime`

## 推荐方案

采用“纯复用现有接口 + 前端推导 + 小幅交互增强”的方案。

原因：

- 不依赖后端改造，交付最快
- 能覆盖用户指出的视觉区块
- 统计口径虽是前端近似值，但明显优于当前静态假数据
- 未来若后端提供专题聚合接口，可平滑替换推导层

## 页面设计

### 1. 拼团氛围卡

该区块保留现有视觉结构，但改为动态字段：

- “今日已成团”
  - 使用当前筛选结果中的 `successGroupCount` 汇总
  - 仅表示当前专题页已加载活动范围内的累计已成团，不宣称全站口径
- “平均省下”
  - 计算当前筛选结果中 `originPrice - price` 的平均值
  - 若原价或拼团价缺失，则跳过该商品
- “还有 N 个团正在冲刺成团”
  - 优先使用主推商品 `activityId` 的 `fetchOngoingGroups()` 结果数量
  - 若没有进行中团，则文案降级为“当前暂无可直接参团，去发起一团”
- 次级说明
  - 展示当前筛选结果中的 `successGroupCount` 总和
  - 不再保留静态增长文案

### 2. 主推团长款

该区块继续以当前筛选结果第一条商品作为主推位，但所有展示信息动态化：

- 标题、图片、拼团价、原价：来自主推商品
- 团长推荐副文案：
  - 优先根据主推商品标签和团时效信息拼接
  - 若缺失，则退化为简洁的活动提示文案
- “已有 X 人拼成，当前 Y 个团可直接加入”
  - `X` 使用商品 `soldQuantity` 或已有映射文案
  - `Y` 使用该商品进行中团数量
- 成功率：
  - 采用 `successGroupCount / (successGroupCount + groupCount)` 的近似值
  - 若分母为 0，则不显示夸张百分比，使用默认提示
- “立省”
  - 使用 `originPrice - price`
- 底部提示：
  - 若有进行中团，显示“当前有 Y 个团正在拼”
  - 若无进行中团，显示“当前暂无可直接参团，支持立即开团”
- CTA：
  - 统一进入该商品详情页的拼团模式

### 3. 列表卡片

列表卡片保留现有 3 条展示上限，但去除写死描述：

- badge：
  - 优先根据 `minPeople` 生成“2人团 / 3人团”
  - 若存在业务标签，则优先使用标签
- 标签：
  - 从 `tags` 中提取最多 2 个
  - 若缺失，则按成团进度自动补“成团率高”“差 X 人成团”等文案
- 热度 / 进度条：
  - 使用与主推卡一致的成功率近似值计算
- CTA：
  - 统一为“去拼团”或“立即参团”
  - 行为统一跳转拼团详情页

### 4. 导航分组

导航逻辑保持现有实现：

- 后端返回 `navTabs` 时优先使用后端配置
- 否则以前端 `minPeople` 做分桶
- 导航切换后，氛围卡、主推位、列表区全部跟随当前筛选结果更新

## 交互设计

### 1. 去捡漏 / 立即参团 / 去拼团

统一跳转到商品详情页，并携带：

- `spuId`
- `orderType=group_buy`
- `groupBuyId=activityId`

说明：

- 当前不承诺透传具体 `groupNo` 实现“直入某个团”
- 但页面文案和入口保持“立即参团”语义，以减少用户理解成本

### 2. 进行中团读取策略

- 页面初次加载时先拉商品列表
- 以当前主推商品的 `activityId` 再拉进行中团列表
- 切换导航时，如主推商品变更，则重新拉对应活动的进行中团
- 若请求失败，不阻断页面主体渲染，只回退到无进行中团状态

## 降级策略

### 1. 无商品

- 保留当前“暂无拼团活动”空态

### 2. 有商品，无进行中团

- 氛围卡显示“暂无可直接参团”
- 主推卡仍正常展示商品
- CTA 退化为引导开团

### 3. 数据字段缺失

- 无图：使用现有卡片占位背景
- 无标签：自动生成通用业务标签
- 原价缺失：不展示划线价与“立省”
- 统计值无法计算：显示中性文案，不显示夸大数字

## 实现边界

主要改动文件预计为：

- `miniprogram/src/pages/promotion/group-buy/index.tsx`
- `miniprogram/src/pages/promotion/group-buy/index.scss`
- `miniprogram/src/services/promotion/groupBuy.ts`（仅在字段适配不足时小幅补充）
- 对应页面测试文件（如项目当前具备页面测试模式）

实现中应尽量把“数据推导函数”和“视图渲染”分离，避免页面组件继续堆积内联硬编码逻辑。

## 测试设计

至少覆盖以下场景：

1. 商品列表返回正常，统计卡能正确展示动态汇总值
2. 主推商品切换后，进行中团数据会随之刷新
3. 无进行中团时，页面展示正确降级文案
4. 导航分组切换后，主推区和列表区同步更新
5. 原价缺失、标签缺失、图片缺失时页面不报错
6. CTA 跳转参数包含 `spuId`、`orderType=group_buy`、`groupBuyId`

## 风险

- 现有接口字段命名存在兼容性映射，前端需继续做容错
- “今日已成团”“成功率”等指标是当前加载数据口径，不应伪装成全站精确数据
- 若后端返回的 `activityId` 与详情页实际需要的 `groupBuyId` 存在偏差，需在实现时实测确认

## 结论

本次改造采用前端复用现有接口的数据驱动方案，以最低依赖成本把拼团专题页从“静态专题壳”升级为“可用的动态运营页”。在不改动整体视觉的前提下，优先解决用户指出的 3 个核心区块，并保证空态、跳转和指标展示都可信可维护。
